# Column-delete / column-insert demotion-then-shift quadratic blowup investigation

> Authored from a read-only `plan` agent investigation (gpt-5.5), reviewed and
> materialized by PM. Anchors every claim in code with file:line refs.

## 1. Reproduction & timing data

### Scenario definitions

The medium corpus cases are generated by:

- `crates/formualizer-bench-core/src/scenarios/s034_family_with_column_insert.rs`
  - rows by scale: small `1_000`, medium `10_000`, large `50_000` at lines 25-31.
  - fixture has one value column and three formula columns at lines 48-52:
    - `A{r}` value
    - `=A{r}+1`
    - `=A{r}*2`
    - `=A{r}-3`
  - medium therefore has 10k value vertices plus 30k formula placements if/when materialized to legacy graph.
  - edit plan is five cycles at lines 77-82.
  - column insert positions are `[3, 2, 5, 1, 4]` at lines 89-91.

- `crates/formualizer-bench-core/src/scenarios/s035_family_with_column_delete.rs`
  - rows by scale: small `1_000`, medium `10_000`, large `50_000` at lines 23-29.
  - fixture has one value input column, five formula columns, and five buffer value columns at lines 46-59:
    - formula columns:
      - `=A{r}+1`
      - `=A{r}*2`
      - `=A{r}-3`
      - `=A{r}+B{r}`
      - `=C{r}+D{r}`
    - buffer value columns `7..=11`
  - medium therefore has 50k formula placements plus 60k value cells.
  - edit plan is five cycles at lines 77-82.
  - every delete removes column `7` at lines 93-100.

The probe runner records per-phase wall time in `PhaseMetrics::finish` at `crates/formualizer-bench-core/src/instrumentation/mod.rs:78-123`. It records FormulaPlane active span count via `introspect_engine` at `instrumentation/mod.rs:126-142`, using `workbook.engine().baseline_stats()`.

The probe runner applies edits and then evaluates after every edit in `crates/formualizer-bench-core/src/bin/probe-corpus.rs:331-353`.

### Hard timing observations supplied for this investigation

Medium-scale corpus run, 10k rows.

#### `s035-family-with-column-delete`

- Off mode: each `delete_columns` phase completed in **138-141 ms**.
- Auth mode: did not finish after **1400 s**; it hung in the edit phase after first demotion.

The edit in this scenario is always `delete_columns("Sheet1", 7, 1)`, from `s035_family_with_column_delete.rs:93-100`.

#### `s034-family-with-column-insert`

Auth mode timings:

- `phase_load`: **240 ms**
- `phase_first_eval`: **7.5 ms**, with **3 active spans**
- `phase_edit_0`: **25,386 ms**; first insert, demotion happens here, spans `3 -> 0`
- `phase_recalc_0`: **16.5 ms**
- `phase_edit_1`: **264 ms**
- `phase_edit_2`: **175 ms**
- `phase_edit_3`: **410,333 ms**; about **410 seconds** for one column insert
- `phase_edit_4`: **247 ms**

Key line refs explaining why `edit_3` is special:

- The insert positions are `[3, 2, 5, 1, 4]` at `s034_family_with_column_insert.rs:89-91`.
- `edit_3` inserts before external column `1`.
- `Engine::insert_columns` converts to zero-based `before0 = before.saturating_sub(1)` at `crates/formualizer-eval/src/engine/eval.rs:3806`, so `edit_3` uses `before0 = 0`.
- `ReferenceAdjuster::adjust_cell_ref` shifts any non-absolute column reference whose zero-based column is `>= before` at `crates/formualizer-eval/src/engine/graph/editor/reference_adjuster.rs:133-148`.
- Therefore `edit_3` is the first post-demotion edit that changes every relative `A{row}` formula reference, because `A` is column `0` and `before0 = 0`.

The observation that `phase_edit_3` occurs after spans are already zero is decisive:

- `Engine::demote_spans_for_structural_op` returns immediately when `active_span_refs()` is empty at `crates/formualizer-eval/src/engine/eval.rs:3630-3633`.
- After `phase_edit_0`, the spans have been removed in the demotion block at `eval.rs:3714-3726`.
- Therefore `phase_edit_3` is not spending 410 s in FormulaPlane demotion. It is in the legacy structural editor path.

## 2. Root cause hypotheses

### H1 — evidenced: `move_vertex -> update_edge_coord` has an O(V) vertex-id lookup, making column shifts O(M * V)

Evidence:

- `VertexEditor::insert_columns` collects vertices to shift at `crates/formualizer-eval/src/engine/graph/editor/vertex_editor.rs:1130-1140`.
- It then calls `self.move_vertex(id, new_coord)?` once per moved vertex at `vertex_editor.rs:1149-1160`.
- `VertexEditor::delete_columns` does the same for shifted survivors at `vertex_editor.rs:1263-1287`.
- `VertexEditor::move_vertex` calls `self.graph.update_edge_coord(id, new_coord)` at `vertex_editor.rs:764-765`.
- `DependencyGraph::update_edge_coord` delegates to `self.edges.update_coord(id, coord)` at `crates/formualizer-eval/src/engine/graph/mod.rs:3460-3463`.
- `CsrMutableEdges::update_coord` uses `self.vertex_ids.iter().position(|&id| id == vertex_id.0)` at `crates/formualizer-eval/src/engine/delta_edges.rs:606-612`.

That `.position(...)` is a full linear scan of the edge-cache vertex-id array per moved vertex.

Cost:

- Let `V` = graph vertices known to `CsrMutableEdges`.
- Let `M` = vertices moved by the structural operation.
- Current `update_edge_coord` cost is O(V) per moved vertex.
- Current shift coordinate-update cost is O(M * V).
- For medium cases with tens of thousands of vertices, this is hundreds of millions to billions of integer comparisons per edit.

This is the most direct quadratic hot-path bug in the actual move path.

This also explains why a post-demotion sheet can become much slower to shift: demotion materializes many FormulaPlane placements as graph vertices before later structural edits. Once those placements exist in `CsrMutableEdges::vertex_ids`, every moved vertex pays a linear search across the enlarged edge-coordinate array.

### H2 — evidenced: `insert_columns` / `delete_columns` do full-graph formula AST adjustment and allocate debug strings for every formula vertex

Evidence:

- `VertexEditor::insert_columns` collects every formula vertex in the entire workbook via `self.graph.vertices_with_formulas().collect()` at `vertex_editor.rs:1172`.
- It then retrieves and adjusts every formula AST at `vertex_editor.rs:1174-1178`.
- It compares old/new formulas using two debug-string allocations:
  - `format!("{ast:?}") != format!("{adjusted:?}")` at `vertex_editor.rs:1178`.
- `VertexEditor::delete_columns` has the same pattern:
  - all formula vertices at `vertex_editor.rs:1298`
  - debug-string compare at `vertex_editor.rs:1303`.

This is unconditional across all formula vertices, not restricted to formulas that reference the affected column range or affected sheet.

Cost:

- Let `N` = formula vertices in workbook.
- Let `A` = average AST size / debug-rendered size.
- Current formula adjust scan is O(N * A) AST traversal plus O(N * debug_string_size) allocation and formatting.
- When references actually change, each changed formula also calls `DependencyGraph::update_vertex_formula` at `vertex_editor.rs:1188` / `1309`, which removes/rebuilds dependency edges at `graph/mod.rs:3543-3598`.

This explains the `s034` variance:

- `edit_1`, `edit_2`, and `edit_4` insert before columns `2`, `5`, and `4`; formulas mostly reference `A`, so `A` is before the insertion point and references do not change.
- `edit_3` inserts before column `1`, which becomes `before0 = 0`; every relative `A{row}` reference changes.
- Therefore `edit_3` triggers both the O(M * V) move path and a high-volume formula-update path.

### H3 — evidenced: `vertices_in_sheet` scans the whole vertex store, not a sheet index

Evidence:

- `DependencyGraph::vertices_in_sheet` is implemented as `self.store.all_vertices().filter(...)` at `crates/formualizer-eval/src/engine/graph/mod.rs:3531-3536`.
- `VertexStore::all_vertices` iterates every allocated vertex id from `0..len` at `crates/formualizer-eval/src/engine/vertex_store.rs:441-443`.

Cost:

- O(V_store) for each collection pass.
- `insert_columns` does one such scan to collect shifted vertices.
- `delete_columns` does two such scans: one for deleted vertices at `vertex_editor.rs:1249-1258`, one for shifted vertices at `vertex_editor.rs:1263-1273`.

This is linear and not sufficient by itself to explain 410 s, but it is on the hot path.

### H4 — evidenced but secondary: `vertices_in_sheet` includes tombstoned vertices

Evidence:

- `DependencyGraph::vertices_in_sheet` filters with `self.vertex_exists(id)` at `graph/mod.rs:3533-3535`.
- `DependencyGraph::vertex_exists` delegates to `VertexStore::vertex_exists` at `graph/mod.rs:3116-3118`.
- `VertexStore::vertex_exists` checks only whether the id maps to an allocated index at `vertex_store.rs:433-435`.
- It does not check `!is_deleted`.
- The active check exists separately as `VertexStore::vertex_exists_active` at `vertex_store.rs:438-439`.
- `remove_vertex` tombstones vertices via `self.graph.mark_deleted(id, true)` at `vertex_editor.rs:711`.

Cost / risk:

- Repeated delete operations leave tombstones in the store.
- Later `vertices_in_sheet` scans include tombstoned vertices.
- `move_vertex` accepts tombstoned ids because `vertex_exists` returns true.
- This can inflate `M` and risks shifting deleted vertices.
- This is more relevant to repeated delete scenarios than to s034 insert.

I do not rank this as the primary 410 s cause because s034 does not delete vertices before `edit_3`.

### H5 — evidenced: `bulk_set_formulas_with_plans` does register demoted vertices in the normal coordinate indexes

Evidence that demoted vertices are registered:

- `bulk_set_formulas_with_plans` creates target vertices through `self.get_or_create_vertex(&addr, ...)` at `crates/formualizer-eval/src/engine/graph/mod.rs:2355-2359`.
- `get_or_create_vertex`:
  - checks `cell_to_vertex` at `graph/mod.rs:2188-2190`,
  - allocates the vertex at `graph/mod.rs:2194`,
  - adds it to edge coordinates at `graph/mod.rs:2197`,
  - adds it to `sheet_indexes` at `graph/mod.rs:2200-2201`,
  - inserts `cell_to_vertex` at `graph/mod.rs:2204`.
- `bulk_set_formulas_with_plans` inserts the formula AST id into `vertex_formulas` at `graph/mod.rs:2370`.
- `vertices_with_formulas()` reads `vertex_formulas.keys()` at `graph/mod.rs:3543-3545`.

Conclusion:

- Demoted vertices are correctly indexed.
- The relevant difference is purely the number/timing of newly materialized graph vertices after demotion, which interacts badly with the O(V) `update_edge_coord` lookup.

### H6 — suspected but not primary: RSS allocator pressure amplifies variance after demotion

Allocator variance does not explain the deterministic spike at `edit_3`; the column position and all-reference update do.

## 3. Code-path analysis

### 3.1 Direct `Engine::insert_columns` and `Engine::delete_columns`

File refs:

- `Engine::insert_columns`: `crates/formualizer-eval/src/engine/eval.rs:3795-3821`
- `Engine::delete_columns`: `eval.rs:3825-3850`

`Engine::insert_columns` hot path:

1. Resolve sheet id at `eval.rs:3801-3805`.
2. Demote FormulaPlane spans at `eval.rs:3807`.
3. Convert to zero-based `before0` at `eval.rs:3808`.
4. Create `VertexEditor::new(&mut self.graph)` and call `editor.insert_columns(...)` at `eval.rs:3811-3814`.
5. Mirror Arrow storage at `eval.rs:3816-3819`.
6. Record FormulaPlane structural change at `eval.rs:3820`.
7. Mark topology edited at `eval.rs:3821`.

### 3.2 `Engine::demote_spans_for_structural_op`

File refs: `crates/formualizer-eval/src/engine/eval.rs:3614-3738`.

After demotion, active spans are zero and later calls return at `eval.rs:3630-3633`.

This path explains `phase_edit_0 = 25,386 ms`, not `phase_edit_3 = 410,333 ms`.

### 3.3 `VertexEditor::insert_columns`

File refs: `crates/formualizer-eval/src/engine/graph/editor/vertex_editor.rs:1114-1223`.

Cost components:

- `vertices_in_sheet`: O(V_store).
- Shift loop: currently O(M * V_edges) due to `update_edge_coord` linear scan.
- Formula adjustment: O(N * A) traversal + O(N * debug_string_size) allocations from `format!`.

`change_logger`:

- In direct benchmark mode, `VertexEditor::new` gives `change_logger: None` at `vertex_editor.rs:219-225`.
- Therefore the logger overhead is not relevant to probe-corpus measurements.

### 3.4 `VertexEditor::move_vertex`

File refs: `crates/formualizer-eval/src/engine/graph/editor/vertex_editor.rs:744-772`.

Per-call cost:

- `update_edge_coord`: currently O(V_edges) — this is the quadratic source.
- `mark_dependents_dirty`:
  - calls `get_dependents` at `graph/mod.rs:3431-3433`.
  - if edge delta op_count is zero, `get_dependents` uses reverse edges at `graph/mod.rs:3305-3308`.
  - if edge delta op_count is non-zero, `get_dependents` scans `cell_to_vertex` at `graph/mod.rs:3277-3304`, O(V_cell_map).

Important nuance:

- `CsrMutableEdges::update_coord` calls `delta.mark_dirty()` at `delta_edges.rs:611`.
- `delta.mark_dirty()` sets `coord_changed`, not `op_count`, at `delta_edges.rs:375-377`.
- `get_dependents` fallback checks `self.edges.delta_size() > 0` at `graph/mod.rs:3280`, and `delta_size()` returns `op_count` at `delta_edges.rs:523-525`.
- Therefore coordinate-only moves do not trigger the `get_dependents` full-scan fallback.
- The evidenced O(M * V) problem is `update_edge_coord`, not `mark_dependents_dirty`, for coordinate-only shift loops.

## 4. Proposed fix

Apply unconditionally. No feature flag. No EvalConfig toggle. No fallback to legacy. No AST arena mutation.

The fix has two tightly scoped pieces, both in the existing legacy structural-op path.

### 4.1 Make `CsrMutableEdges::update_coord` O(1)

Change:

- Add a side index to `CsrMutableEdges`:

```rust
vertex_pos: FxHashMap<u32, usize>
```

Maintain it in every constructor/mutator:

- `new`: empty map.
- `with_coords`: populate for generated ids.
- `add_vertex`: insert `vertex_id -> idx`.
- `add_vertices_batch`: insert each new id with its pushed index.
- `build_from_adjacency`: rebuild map from supplied `vertex_ids`.
- Any future method that replaces `vertex_ids` must rebuild the map.

Then implement:

```rust
pub fn update_coord(&mut self, vertex_id: VertexId, new_coord: AbsCoord) {
    if let Some(&pos) = self.vertex_pos.get(&vertex_id.0) {
        self.coords[pos] = new_coord;
        self.delta.mark_dirty();
    }
}
```

Add `debug_assert_eq!(vertex_ids[pos], vertex_id.0)` in `update_coord` for safety.

Expected complexity change:

- Before: O(M * V_edges) per structural shift.
- After: O(M) for edge-coordinate updates.

### 4.2 Replace debug-string AST compare with change-aware reference adjustment

Add a change-aware API to `ReferenceAdjuster`:

```rust
pub fn adjust_ast_if_changed(&self, ast: &ASTNode, op: &ShiftOperation) -> Option<ASTNode>
```

Implementation details:

- Traverse the AST.
- For `Reference`, call existing `adjust_reference`.
- Compare `ReferenceType` directly via `PartialEq`. Verified: `ReferenceType` derives `PartialEq` at `crates/formualizer-parse/src/parser.rs:240`, `ASTNode` derives `PartialEq` at `parser.rs:1722`.
- Return `None` if no reference changed.
- Return `Some(adjusted_ast)` only when a subtree actually changed.
- Preserve existing semantics: handle the same AST node kinds currently adjusted by `adjust_ast` (`Reference`, `BinaryOp`, `UnaryOp`, `Function`).

Then update editor loops:

```rust
if let Some(adjusted) = adjuster.adjust_ast_if_changed(&ast, &op) {
    ...
    self.graph.update_vertex_formula(id, adjusted)?;
}
```

Eliminates O(N * debug_string_size) allocations.

### Alternatives considered and rejected

1. Feature flag / EvalConfig toggle — rejected. Fix must be unconditional.
2. Ignoring or expected-failing s034/s035 — rejected. Real bug.
3. Fallback to legacy by disabling FormulaPlane demotion — rejected. Pathology is in legacy path.
4. AST arena mutation in place — rejected. Would violate non-mutating invariant.
5. Speculative cache of adjusted formulas — rejected. No cache key, no eviction.
6. Rewrite structural-op machinery — rejected. Two small targeted fixes suffice.
7. Parallelization — rejected. Bottleneck is algorithmic complexity.
8. Use sheet indexes for `vertices_in_sheet` as the main fix — rejected. Linear, not quadratic.
9. Filter active vertices in `vertices_in_sheet` as the main fix — rejected for s034 (no tombstones); separate concern.

## 6. Risks and rollback

### Risk 1: stale `vertex_pos` map

Add debug assertions everywhere we mutate `vertex_ids` to assert `vertex_ids[vertex_pos[vid]] == vid`.

Rollback: revert the field; restore `.position(...)`. Isolated to `delta_edges.rs`.

### Risk 2: change-aware AST adjustment misses a semantic change

Detection: existing reference_adjuster tests, structural tests, new repeated-insert test, probe-corpus invariants.

Rollback: revert to `let adjusted = adjuster.adjust_ast(...)` and direct `if ast != adjusted` (still avoiding debug-string compare).

### Risk 3: `PartialEq` differs from debug-string equality

Very unlikely. Both `ASTNode` and `ReferenceType` derive `PartialEq` (verified). `DataStore::reconstruct_ast_node` sets `source_token: None` so debug strings already drop tokens.

### Risk 4: delete tombstone behavior remains a latent issue

Out of scope for this fix. Add to follow-up.
